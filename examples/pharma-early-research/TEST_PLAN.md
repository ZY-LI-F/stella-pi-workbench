# 医药早研靶点评估 E2E 测试方案

## 1. 测试目标

用一个真实、可复跑的 NLRP3 早研任务验证 Stella Pi Workbench 的完整链路：项目级医药 Skills 被 Pi 发现并强制校验，固定 Agent 团队按 Workflow 分工，用户通过页面创建和分发任务，在人工关卡确认范围，最终在 Task Room 与可视化 DAG 中审阅报告、审计结论和来源，再完成显式验收。

决策问题固定为：**是否启动“口服、脑穿透、选择性 NLRP3 小分子抑制剂用于早期帕金森病炎症富集人群”的正式发现项目？** 数据快照为 2026-07-18。

## 2. 测试对象与前置条件

- 示例项目目录是本目录，必须以“受信任项目”打开；否则 Pi 按安全边界不加载 `.pi/skills`。
- 必需 Skills：`target-evidence`、`clinical-landscape`、`target-assessment-report`。
- 固定团队：靶点生物学研究员、临床竞品分析师、靶点策略负责人、证据审计员。
- 固定流程：`早研靶点评估流程`，包含 4 个 Agent 步骤和 2 个人工关卡。
- 真实 Agent E2E 需要 Pi 已配置可调用的模型凭据，并允许访问 Open Targets、Human Protein Atlas、ChEMBL 和 ClinicalTrials.gov。
- 不允许 mock 模型、伪造报告、吞掉 API 错误或在 Skill 缺失时自动改用模板。

## 3. 分层测试矩阵

| 层级 | 目的 | 关键检查 | 是否依赖模型/网络 |
|---|---|---|---|
| 单元测试 | 验证 Skill 契约 | 精确匹配 `skill:<name>`；缺失、禁用、畸形 RPC 响应均显式失败 | 否 |
| 目录/领域测试 | 验证固定编排 | Agent、Team、Workflow 引用完整，领域 Agent 开启 Skills 且列出必需项 | 否 |
| 确定性页面 E2E | 验证用户入口 | 打开受信任示例项目、命令面板发现 Skill、目录展示要求、页面创建任务、选择早研流程、Task Room 显示待分发 DAG | Pi RPC 是；模型否 |
| 真实页面 E2E | 验证业务闭环 | 页面分发、真实 API 取证、两次人工关卡、四个真实 Agent 产物、DAG 状态、报告验收和截图 | 是 |
| 回归 | 保护 Pi 原功能 | typecheck、全部 Vitest、原 Electron E2E、构建 | 视原测试而定 |

## 4. 测试任务输入

任务标题：`NLRP3 早研靶点评估`

任务说明必须包含：

- 靶点：NLRP3；Ensembl：ENSG00000162711；物种：Homo sapiens。
- 模态：口服、脑穿透、选择性小分子抑制剂。
- 首选适应症假设：早期帕金森病中的炎症富集人群。
- 竞争范围：NLRP3 直接抑制剂的人体干预试验；观察性研究和仅下游 IL-1/IL-6 抑制剂单列，不混入直接竞品。
- 证据快照日期、原始数据路径、NCT ID、试验状态更新时间和终止原因必须保留。

验收标准：

1. Open Targets、Human Protein Atlas、ChEMBL、ClinicalTrials.gov 四个官方数据源均有可追溯证据或明确失败。
2. 报告包含靶点身份、人类验证、机制、表达、成药性、安全、转化标志物、竞品表、加权评分、结论、反证条件、90 天补证计划和来源台账。
3. 所有评分展示算术；任何无来源的决策性事实标为缺口，不得补造。
4. 至少覆盖 dapansutrile、DFV890、VTX3232、VTX2735、selnoflast、NT-0796/ruvonoflast、VENT-02、ZYIL1，并保留 terminated/withdrawn 项目。
5. 证据审计员独立复算总分，指出会改变结论的问题。

## 5. 页面 E2E 主路径

1. 用隔离的 Electron user-data 启动应用，并让持久化状态把本目录标为受信任项目。
2. 在命令面板搜索 `target-evidence`，确认其来源为 Skill；依次确认三个 Skill 可发现。
3. 打开“编排目录”：检查四个医药 Agent、`早研靶评小队`、`早研靶点评估流程`，以及每个 Agent 卡片上的 `skill:*` 要求。
4. 点击“新建任务”，填写固定输入，选择“固定流程 → 早研靶评”，创建任务。
5. 打开 Task Room，确认执行目标、任务事实流和“尚无持久化 Run”；截取确定性页面截图。
6. 点击“开始执行”。等待“靶点证据”和“竞品扫描”完成；检查 DAG 两节点成功、原始证据路径存在、Task Room 有两个独立产物。
7. 在“证据范围确认”填写批准说明并点击“批准并继续”。
8. 等待“决策报告”和“证据审计”完成；检查报告章节、NCT ID、来源、评分算术和审计发现。
9. 在“组合评审”批准；随后点击“接受报告”。确认任务进入“已完成”，执行状态为“已报告”，验收状态为“已接受”；截取最终页面截图。

## 6. 必测失败路径

- 删除或改名任一必需 Skill：Agent 在收到模型 prompt 前失败，Task Room 显示缺失的精确 Skill 名称和安装位置建议。
- 把领域 Agent 配置为 `disableSkills=true`：配置错误直接失败。
- 以不受信任项目打开：项目 Skill 不应被悄悄加载，执行因预检失败而终止。
- API 返回非 2xx 或无效 JSON：采集脚本以非零退出码结束并保留错误，不生成 `ok: true`。
- ClinicalTrials.gov 与申办方公告状态冲突：报告以最新注册状态为主，并记录冲突和各自日期。
- 报告缺少来源或评分计算错误：证据审计必须判为会改变结论的发现，人工最终关卡不应批准。

## 7. 通过标准

- `npm run typecheck`、`npm test`、`npm run test:e2e:pharma` 全部通过。
- `npm run test:e2e:pharma:live` 在真实凭据环境中通过，且没有测试跳过、页面异常、RPC 协议错误或未处理的控制台错误。
- 两份原始证据 JSON 包含快照时间与官方源 URL；报告结论可由这些证据和明确引用的原始研究复算。
- 页面完成创建、分发、关卡、DAG 检查、报告阅读和验收；不能用直接改数据库替代业务交互。

## 8. 数据与合规边界

本场景只使用公开数据库和公开临床信息，不处理患者级数据，不提供诊疗建议。报告用于研发组合优先级测试。快照结论会随试验状态变化；复跑时必须生成新快照，不能覆盖旧日期后继续沿用旧结论。
